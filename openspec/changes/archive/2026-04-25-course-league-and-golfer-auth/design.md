## Context

The scaffold is fully wired (postgres → api → web, migrations run on startup). The domain model is designed in `docs/data-model.md`. This change implements the foundational slice of that model — the entities that exist before any gameplay happens — plus a working mock auth system and a "my profile" page.

Real Auth0 wiring is deferred to a later proposal. The mock provider is the only working auth implementation here, deliberately, so the full UI flow is exercisable without any external dependency.

## Goals / Non-Goals

**Goals:**
- EF Core entities and a migration for `Course`, `TeeBox`, `Hole`, `League`, `LeagueConfiguration`, `Season`, `LeagueMembership`, and `Golfer`
- Audit fields (`CreatedAt`, `UpdatedAt`) on every entity; soft-delete (`ArchivedAt`) on `Golfer`, `Season`, `LeagueMembership`; `CreatedBy`/`UpdatedBy` on commissioner-editable entities per `docs/data-model.md`
- Unique constraints on `Golfer(CourseId, Email)` and `Golfer(CourseId, ExternalAuthId)`
- UUID primary keys throughout, generated by the database
- SQL seed template to bootstrap a course, league, season, and golfer accounts
- `IAuthProvider` abstraction with `MockAuthProvider` (working) and `Auth0AuthProvider` (stub)
- "Log in as" UI: a dev-only dropdown listing all seeded golfers; selecting one sets the active mock session
- Golfer resolution middleware: reads the mock session cookie, looks up the golfer, attaches to request context
- `GET /me` returning the authenticated golfer's profile and their active league memberships
- "My profile" Next.js page displaying the golfer's name, course, and active league memberships
- Ping spike cleanup: remove `Ping` entity, migration, and `/ping` endpoint/page

**Non-Goals:**
- Real Auth0 JWT validation — `Auth0AuthProvider` throws `NotImplementedException`
- `@auth0/nextjs-auth0` SDK — no Auth0 frontend dependency in this proposal
- Golfer self-registration
- Course admin role
- Any UI for managing courses, leagues, or seasons

## Decisions

### D1: UUID primary keys via `gen_random_uuid()` database default
All PKs are `uuid`, generated by PostgreSQL using `gen_random_uuid()`. EF Core entities use `Guid` with `HasDefaultValueSql("gen_random_uuid()")`.

**Why over sequential integers:** UUIDs avoid enumerable IDs in URLs and are consistent with the data model design.

### D2: Auth abstraction via `IAuthProvider`
`IAuthProvider` exposes one method: `ValidateTokenAsync(string token) → Task<AuthResult?>` where `AuthResult` carries `ExternalAuthId` and `Email`. Two implementations ship in this proposal:

- `MockAuthProvider` — reads a `mock-golfer-id` cookie set by the "log in as" UI. Returns an `AuthResult` with the golfer's id as `ExternalAuthId`. No token, no JWT, no signature.
- `Auth0AuthProvider` — throws `NotImplementedException`. Exists so the abstraction is wired and the next proposal can drop in a real implementation without touching the middleware or route handlers.

The active provider is selected by `AUTH_PROVIDER` env var (`mock` | `auth0`), defaulting to `mock`. A startup guard throws if `AUTH_PROVIDER=auth0` since that implementation is not yet functional.

**Why cookie instead of a JWT for the mock:** A cookie-based mock is simpler to implement in the "log in as" UI — no token generation required. The middleware reads the golfer id directly. The abstraction still looks like a token provider to the rest of the system.

### D3: "Log in as" UI
A dev-only Next.js page at `/dev/login` renders a dropdown of all golfers fetched from `GET /dev/golfers` (dev-only endpoint). Selecting a golfer POSTs to `POST /dev/login` which sets the `mock-golfer-id` cookie and redirects to `/me`. Both the endpoint and the page are only available when `AUTH_PROVIDER=mock`.

**Why a page instead of a curl command:** A UI is faster to use during development and more accessible to contributors who aren't comfortable with curl. The endpoint is still usable via curl for scripting.

### D4: Golfer resolution middleware
`GolferContextMiddleware` runs after routing:
1. Calls `IAuthProvider.ValidateTokenAsync` (which for mock reads the cookie)
2. Looks up `Golfer` by `ExternalAuthId`
3. Stores the resolved `Golfer` on `HttpContext.Items["Golfer"]`

Routes that require auth call `HttpContext.RequireGolfer()` which returns 401 if no golfer is attached.

No claim-by-email flow in this proposal — golfer ids are set directly by the mock UI. Claim-by-email belongs with real Auth0 wiring.

### D5: `GET /me` includes active league memberships
The endpoint returns the golfer's profile plus a list of their active `LeagueMembership` records (not archived, within a non-archived `Season`), each including the league name and season year. This is the data the "my profile" page needs.

### D6: EF Core entity configuration
All entity configuration lives in `IEntityTypeConfiguration<T>` classes in `api/Data/Configurations/`. No data annotations on the entities.

Column naming: snake_case via `UseSnakeCaseNamingConvention()` (EFCore.NamingConventions package) applied globally.

### D7: Ping spike cleanup
The `Ping` entity, migration, `/ping` endpoint, and `/ping` Next.js page are removed. The database volume must be recreated (`docker compose down -v`).

## Risks / Trade-offs

- **Mock cookie has no expiry or security** → Intentional for dev; the mock provider must never be registered in production. Startup guard enforces this.
- **`Auth0AuthProvider` stub will cause a loud failure if selected** → By design — makes it obvious the real implementation is missing rather than silently doing nothing.
- **Database volume must be dropped** → Loses seeded data. Acceptable at this stage; the seed template makes it easy to re-seed.
- **`EFCore.NamingConventions` is a new package dependency** → Lightweight and well-maintained. Can be removed and replaced with explicit `HasColumnName` calls if needed.

## Open Questions

- Should the `mock-golfer-id` cookie be scoped to the API domain or the web domain? Since the web makes API calls server-side, the cookie needs to be forwarded — evaluate during implementation whether a shared cookie domain or a session token stored in the web layer is cleaner.
